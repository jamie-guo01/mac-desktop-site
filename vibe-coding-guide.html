设计师的 Vibe Coding
完整工作流

设计师不需要学会"写代码"，而是学会"指挥 AI 写代码"。这份指南将帮助你从零建立一套可复用、可交付的 AI 辅助开发工作流。

独立完成前端 UI 开发并交付
设计到开发交付周期缩短 50%
和开发建立高效协作模式
开始阅读 直接看 Prompt 模板
黄金原则

UI/UX 设计开发阶段只用 Mock 数据，不碰业务逻辑。UI 与业务逻辑分离，设计师负责前端界面和交互，开发负责接口和业务。

适合使用的场景

  • 有明确设计稿的 UI 开发（新页面、新组件、改版）
  • 确定性高的需求，交互和视觉已经定稿
  • 团队愿意接受设计师提交的前端代码

不适合的场景

  • 探索性原型或概念验证（需求还没定）
  • 需要深度业务逻辑的功能模块

1-3 建立 CLAUDE.md 项目上下文文件

在项目根目录创建 CLAUDE.md 文件——这是你和 AI 的"约定书"：

# 项目说明

## 技术栈
- 框架：React 18 + TypeScript
- 样式：Tailwind CSS
- 组件库：shadcn/ui
- 包管理：pnpm

## 文件结构
- 页面文件放在 src/pages/
- 组件文件放在 src/components/
- Mock 数据放在 src/mock/

## 开发约定
- 所有 Mock 数据必须有 TypeScript 类型定义
- 业务逻辑用 TODO 注释占位，不自行实现
- 不修改 src/api/ 目录下的任何文件

## 禁止事项
- 不调用真实 API
- 不修改路由配置文件
- 不改动已有业务组件
为什么重要：AI 在长时间对话中会"忘记"你早期说的规范，写在文件里它每次都会重新读取，相当于一份永不过期的指令。

1-4 启动项目 & 预览效果

环境装好后，先把项目跑起来。后续开发阶段的每一次修改都需要在浏览器里看效果，这是最基本的操作循环。

启动项目三步走

# ① 在编辑器中打开终端（Cursor / VSCode 底部的 Terminal 面板）
# 快捷键：⌃` (Control + 反引号)

# ② 安装项目依赖（仅首次 clone 后需要执行一次）
npm install
# 如果团队用 pnpm，则用：pnpm install

# ③ 启动开发服务器
npm run dev

看到这样的输出就说明成功了

  VITE v5.x.x  ready in 320 ms

  ➜  Local:   http://localhost:5173/
  ➜  Network: use --host to expose

按住 ⌘ Cmd 点击终端里的链接，浏览器会自动打开页面。

开发循环：修改 → 保存 → 自动刷新

AI 修改代码
在编辑器中编辑文件
保存文件
⌘S 保存（通常自动保存）
浏览器自动刷新
切到浏览器即可看到最新效果
现代前端项目都支持热更新（Hot Reload）——代码一保存，浏览器会自动刷新页面显示最新效果，不需要手动刷新。这就是你在开发阶段反复调整 UI 的基本操作循环。

启动遇到问题？

报错 command not found: npm
→ Node.js 没装好，回到 1-2 重新安装
报错 EACCES permission denied 或依赖安装失败
→ 把完整报错信息复制给 AI，说"帮我解决这个报错"
端口被占用 Port 3000 is already in use
→ 关掉之前的终端窗口重新启动，或让 AI 帮你换个端口
重要：开发服务器启动后不要关闭终端窗口，它需要一直运行。如果不小心关了，重新执行 npm run dev 即可。

Phase 2 · 核心阶段

开发阶段

2-1 让 AI 写 PRD 和技术方案

不要上来就让 AI 写代码，先让它理清思路：

我是一个 UI/UX 设计师，我要开发一个[模块名称]页面。

设计稿描述：
- [描述页面主要内容]
- [描述主要交互：如筛选、分页、弹窗]
- [描述数据展示：如列表、卡片、图表]

请帮我：
1. 整理这个页面的功能点清单（PRD）
2. 分析需要哪些组件
3. 规划开发顺序（从简单到复杂）
4. 列出需要 Mock 的数据结构

注意：这个阶段只做静态 UI，不涉及真实接口。

拿到方案后，确认没有遗漏再进入开发。

2-2 准备 Mock 数据 关键步骤

在写任何 UI 之前，先让 AI 生成 Mock 数据：

根据上面的功能分析，请帮我生成这个页面需要的 Mock 数据。

要求：
1. 用 TypeScript 定义类型（interface）
2. 生成 10 条以上的测试数据，数据要多样化
3. 包含边界情况（超长文字、空值、特殊字符）
4. 文件放在 src/mock/[模块名].ts
贴近真实接口
后期替换时改动量小
多样化数据
避免测试数据千篇一律
包含边界情况
提前暴露 UI 布局问题

2-3 静态 UI 开发

推荐从大到小、从框架到细节的顺序：

第一轮
搭框架

实现整体布局，用 Mock 数据填充，先不加交互

第二轮
补组件

逐个实现具体组件：筛选栏、数据表格、弹窗等

第三轮
加交互

展开/折叠、筛选、动画，数据变化仍在 Mock 范围内

每完成一个小功能就在浏览器里看效果，不要等全部写完再看。

Prompt 示例 — 第一轮搭框架

请根据设计稿实现 [页面名] 的整体布局：
1. 左侧 240px 侧边栏 + 右侧内容区
2. 顶部搜索栏 + 筛选区域
3. 主体是一个数据列表（用 Mock 数据填充 5 条）
4. 先不做交互，只做静态布局
技术栈：[React/Vue + 组件库]

2-4 交互打磨

视觉和交互都完成后，让 AI 做自检：

请检查当前代码，找出以下问题并修复：
1. 是否有硬编码的字符串（应该用变量）
2. 是否有 console.log 未清理
3. 是否有未处理的 TODO
4. 响应式布局是否有问题（移动端/宽屏）
5. 是否有明显的性能问题（如列表没有 key）

同时做极端情况测试：

Mock 数据改成空数组，看空状态处理
文字改成超长字符串，看是否溢出
数字改成 0 或负数，看显示异常

2-5 业务逻辑占位

UI 完成后，主动标记所有需要接入真实逻辑的地方：

// TODO: 获取用户列表 - 需要调用 getUserList 接口
// 当前：使用 Mock 数据
// 替换为：调用 src/api/user.ts 中的 getUserList 方法

3-2 推送独立分支

每个功能 / 页面开独立分支，命名建议：

feat/[你的名字缩写]-[功能名]-ui

例如：feat/hsf-seo-audit-ui

Git 速查 — 设计师常用命令

# 创建并切换到新分支
git checkout -b feat/hsf-seo-audit-ui

# 查看改了哪些文件
git status

# 提交所有改动
git add . && git commit -m "feat: seo audit page UI"

# 推送到远程
git push origin feat/hsf-seo-audit-ui

不熟悉 Git？直接让 AI 帮你执行命令，说"帮我把代码推送到新分支 feat/xxx"即可。

FAQ

常见问题

AI 生成的代码报错了怎么办？
把报错信息完整复制给 AI，说"帮我修复这个报错"，不要自己猜。AI 通常能根据报错信息精准定位问题。
AI 改了我不让它改的文件怎么办？
在 CLAUDE.md 中明确写"禁止修改 xxx 文件"，并在每次对话开始时提醒。用 Git 可以随时回退误改的文件。
生成的 UI 和设计稿差很多怎么办？
截图对比，告诉 AI"这是设计稿截图，这是当前效果，请调整以下部分：..."。视觉对比比文字描述更高效。
如何让 AI 产出符合产品风格的 UI？

核心思路：把"产品风格"变成 AI 能读懂的结构化规则，而不是靠每次 Prompt 描述。在 CLAUDE.md 或 .cursor/rules/ 里写死四件事：

1

对齐 AI 审美

用一段话描述产品调性 + 参考产品，例如「类似 Linear 的紧凑感」「接近 Notion 的留白」。审美氛围词一写清楚，AI 的默认风格会立刻靠近你的产品。

2

锁死 Design Token

明确规定：颜色、间距、字号只能用 Design Token（CSS 变量 / 主题变量），不准写魔法数字和裸色值。避免 AI 自己"调色"偏离品牌。

3

强制复用组件库

下死命令「不允许引入新的 UI 库，必须用项目现有的 xxx（如 shadcn/ui、Ant Design、自研组件库）」。新增组件前先检索，杜绝重复造轮子。

4

限制写法边界

不允许写内联 style，除非是动态计算的值。样式统一走 Tailwind / CSS Module / 主题系统，保证项目整体风格可维护、可追溯。

一句话总结：Prompt 里强约束，规则里定边界。能给 AI 一个「沙盒」，就不要给它一张「白纸」。

对话太长 AI 开始乱改怎么办？
开一个新对话，把 CLAUDE.md 内容和当前进度重新告知 AI，继续工作。长对话是 AI 质量下降的主要原因。
不知道一个功能该怎么实现怎么办？
先问 AI"这个功能有哪些实现方式，各有什么优缺点"，选一个再让它实现。不要自己猜，让 AI 做技术选型。
担心代码质量不够好，开发不接受怎么办？
记住你交付的是静态 UI 层，不涉及架构决策。只要做到：① 文件结构和命名遵守项目规范；② 所有业务逻辑用 TODO 标注；③ 附带清晰的对接文档 — 开发接手成本极低。前 1-2 次交付主动邀请开发 Review，根据反馈调整，信任很快就会建立起来。
完全不懂 Git，能做 Vibe Coding 吗？
可以。日常只需要 4 个命令：checkout -b（建分支）、status（看状态）、add + commit（提交）、push（推送）。不确定的时候直接告诉 AI"帮我把代码提交并推送到远程"，它会帮你执行。
公司内部推进时常见坑有哪些？怎么应对？

代码质量被开发否定：主动说"这是初稿，帮我看哪里需要改"，而不是"这是我写的代码"。

开发觉得你越权：强调"减少重复劳动"，而不是"替代你"。

承接后出 bug 背锅：提前和 PM / 开发约定好交付物的验收标准。

AI 遗忘早期约定：关键约束写进 CLAUDE.md 文件，不只在对话里说。

设计师不需要学会写代码，而是学会指挥 AI 写代码
